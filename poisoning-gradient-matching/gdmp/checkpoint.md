# Guided Diffusion for Model Purification (GDMP) - Architecture & Workflow Checkpoint

This document serves as the central checkpoint and reference for the GDMP subset purification pipeline. It details the precise workflows, data paths, and logic loops spanning across the codebase, specifically handling isolated 10-case evaluation runs. 

**Whenever interacting with the GDMP module, adhere to the architectural rules and paths defined below.**

---

## 1. High-Level Workflow Overview

The GDMP pipeline aims to evaluate a defensive mechanism (Guided Diffusion) against Gradient Matching poison attacks (e.g., Witches' Brew). Instead of purifying all 50,000 images in CIFAR-10, the "subset" approach extracts only the 500 compromised (poisoned) images, diffuses/purifies them, and merges them back to track the effectiveness of the defense on model retraining.

### Execution Sequence (`run_10_cases.sh`)
The shell script orchestrates the pipeline over 10 isolated configurations:
1. **Attack Generation:** Executes `brew_poison.py` to generate a 1% budget Gradient Matching attack against a specific `--poisonkey` and single target. Saves the full dataset (with injected poisons) to `poisons/case_X`.
2. **Purification & Evaluation:** Calls `gdmp/train_purify.py` to handle the subsequent distillation, merging, and ResNet18 retraining natively.

---

## 2. Directory and Data Path Structure

The pipeline transitions between temporary raw folders, flat extraction folders, and merged training folders to ensure isolated environments for every parameterized run (preventing overwrites during grid search).

### Logging Flow Path
Logs for experiments are strictly categorized by dynamic timestamp identifiers (`EXP_NAME="exp_$(date +%Y%m%d_%H%M%S)"`) natively triggered upon batch script execution.
1. **Experiment Log Root:** `gdmp/exp_{date}_{time}/`
2. **Train & Purify Phase Logs:** `gdmp/exp_{date}_{time}/train_purify/train_purify_case{X}_Tc{Tc}_M{M}.log` (Tracks retraining success vs attack viability per model test).
3. **Global Pipeline Logs:** `gdmp/exp_{date}_{time}/pipeline/pipeline_case{X}...log`

### Data Flow Path
1. **Raw Poison Generation:** `poisons/case_{case_id}/train/`
   - Generated by `brew_poison`. Contains class-separated subdirectories (e.g., `automobile/`, `bird/`) holding all 50,000 images, including the 500 poisoned ones.
2. **Poison Extraction (`flat_poison_dir`):** `poisons/case_{case_id}/poisoned_images/`
   - A flat directory containing *only* the 500 targeted `.png` files. 
3. **Purified Images:** `gdmp/exp_{date}_{time}/purified_images/purified_images_case{case_id}_Tc{Tc}_M{M}/`
   - The output from `diffuse.py`. Contains the 500 images after they have passed through the DDPM reverse process.
4. **Merged Training Dataset:** `gdmp/exp_{date}_{time}/dataset/train_case{case_id}_Tc{Tc}_M{M}/`
   - A full copy of the `poisons/case_{case_id}/train/` folder, but the 500 raw poisoned images are fully overwritten/replaced by the DDPM purified variants form the Purified Images folder.

## 3. Module & Logic Breakdown

### A. `gdmp/train_purify.py` (The Orchestrator)
This is the master python script executed by the batch runner. It dictates the 4-step pipeline:
* **Step 0 (Extraction):** 
  - Restores the exact `forest.Kettle` configuration using the `--poisonkey` string passed by the bash script.
  - Queries `kettle.poison_ids` to retrieve the exact 500 image IDs that were poisoned.
  - Uses `glob.glob` to find these specific images inside `poisons/case_X/train/*/{pid}.png` and copies them into the flat `poisoned_images` directory.
  - *Failsafe:* If 0 images are extracted, it invokes `sys.exit(1)` assuming `brew_poison.py` failed.
* **Step 1 (Purification):** 
  - Calls `diffuse.run_purification_pipeline()` on the flat folder.
* **Step 2 (Merging):** 
  - Calls `merge_dataset.build_purified_dataset()` to create the final merged ImageFolder dataset.
* **Step 3 & 4 (Retraining and Evaluation):** 
  - Initializes a `forest.Victim` model (ResNet18) to the precise vulnerability state (`--modelkey 177672595`).
  - Trains from scratch on the newly purified merged dataset using exact Witches' Brew differentiable augmentations and a strict batch size of 128 (required for gradient matching viability).
  - Evaluates the targeted instance. If the model outputs the intended adversarial class, the attack fooled the network. If the predicted label does NOT match the intended label, GDMP successfully defended the network.

### B. `gdmp/diffuse.py` (The Defense Mechanism)
Executes the Guided Diffusion logic via a pre-trained offline Hugging Face DDPM (`google/ddpm-cifar10-32`).
* Uses `batch_size=64` to prevent GPU memory overflow during processing.
* **Purification Equation (`purify_gdmp_batch`):**
  - Iterates $M$ times.
  - In each iteration, forwards the image directly to timestep $Tc$ using `scheduler.add_noise()`.
  - Performs standard reverse DDPM denoising back to $t=0$, but applies a **Guidance Shift** at each step:
    - Calculates an MSE loss between the current denoised step ($x_t$) and the original poisoned image noised to the same step ($x\_adv_t$).
    - Computes the gradient: `grad_x_t = torch.autograd.grad(loss, x_t)`.
    - Updates the denoised prediction: `x_prev = x_prev - (s_t * sigma_sq * grad_x_t)`.
    - This mathematically constrains the generative model to not stray too far from the original image's underlying clean semantics while shedding the high-frequency poison gradient perturbations.

### C. `gdmp/merge_dataset.py` (The Assembler)
Prepares the data for standard PyTorch `ImageFolder` loaders.
* Uses standard `shutil.copytree` to clone the entire 50,000-image dataset to the new target folder.
* Iterates through all `.png` files in the purified directory, scans the subdirectories of the copied dataset, and directly overwrites the matching localized poison files.

## 4. Offline Environment Constraints (Critical Notice)
The compute nodes running the GDMP pipeline **do not have internet access**. Models cannot be fetched from the `huggingface.co` hub dynamically during runtime. 
- A local cache at `local_ddpm_model/` must always exist.
- Scripts interacting with `diffusers` (e.g., `diffuse.py`) must declare `os.environ["HF_HUB_OFFLINE"] = "1"` to force localized model resolution.
